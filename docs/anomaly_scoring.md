# Anomaly Scoring and App Objective

## 1. Why this document exists

This project evolved from a simple predictive maintenance demo into a more complete **Hybrid AI Predictive Maintenance Lab**.

The current app is not designed to show that a Large Language Model can replace classical predictive maintenance models. Its objective is to demonstrate a practical industrial architecture where each component has a clear role:

- raw vibration data represents the physical signal;
- deterministic physics-informed scoring creates an interpretable diagnostic baseline;
- SQL templates and guards provide safe data access;
- GenAI-style routing helps users query the database in natural language;
- future ML/DL models can be added on top of a traceable baseline;
- human validation remains part of the decision process.

The most important design principle is:

> ML decides where it is strong. LLM explains or routes where it is safe. Agents coordinate. Humans validate.

At the current stage, action plans and anomaly severity are not generated by an LLM. They are derived from deterministic rules and signal-processing features.

---

## 2. Current app objective

The app currently has two main flows.

### 2.1 Monitoring

The monitoring flow shows the latest available condition by equipment up to a selected collection date.

The current monitoring pipeline is:

```text
selected collection date
→ latest measurement per equipment up to that date
→ raw vibration samples
→ time-domain chart
→ FFT spectrum
→ PSD / broadband chart
→ deterministic anomaly scoring
→ rule-based recommended action
```

The monitoring view is meant to answer questions such as:

- Which equipment is currently normal, in attention, or critical?
- Which fault pattern is most likely for each asset?
- What physical signal supports the diagnosis?
- Is the issue more aligned with lubrication/carpet behavior or structural looseness?
- What deterministic action should maintenance prioritize?

### 2.2 Chat GenAI

The Chat GenAI flow allows the user to query the predictive maintenance database using natural language.

The current query pipeline is:

```text
user question
→ Domain Guard
→ keyword semantic router
→ embedding router fallback
→ approved SQL template
→ SQL Guard
→ SQLite result
→ equipment cards
→ deterministic action plan
```

This layer does not currently use an LLM to freely generate SQL. It uses controlled routing and approved SQL templates to preserve safety, traceability, and reproducibility.

## 3. What the anomaly score is

The current `overall_anomaly_score` is a **physics-informed deterministic anomaly score**.

It is not a trained machine learning model.

It is calculated from vibration features extracted from the raw signal. These features are designed to capture two initial diagnostic patterns:

1. carpet / lubrication issue;
2. structural looseness.

The score is interpretable and rule-based. It is intended to serve as a baseline before adding machine learning models such as Isolation Forest, supervised classifiers, autoencoders, CNNs, or LSTMs.

A precise way to describe it is:

> A deterministic physics-informed anomaly scoring baseline derived from time-domain and spectral vibration features.

---

## 4. What the anomaly score is not

At the current stage, the anomaly score is not:

- a supervised classifier;
- an Isolation Forest result;
- a One-Class SVM result;
- an autoencoder reconstruction error;
- a CNN prediction;
- an LSTM sequence anomaly score;
- a calibrated probability from a trained model;
- a replacement for expert vibration analysis;
- a final production-grade maintenance decision.

The project uses the table name `ml_diagnostics` because the architecture is prepared for a future ML layer. However, the current values stored there are still generated by deterministic scoring rules.

This distinction is important for technical credibility.

## 5. Signal-processing pipeline

The seed pipeline generates and stores raw vibration samples. Each measurement has:

- sample index;
- time in seconds;
- velocity in mm/s;
- sampling rate;
- duration;
- RPM;
- time-domain metrics;
- spectral features;
- diagnostic scores.

The conceptual processing flow is:

```text
raw vibration signal
→ remove mean
→ FFT
→ frequency-band energy
→ rotational frequency markers
→ harmonic and subharmonic indicators
→ carpet score
→ looseness score
→ overall anomaly score
→ anomaly probability-like score
→ predicted condition
→ severity
```

---

## 6. Carpet / lubrication issue score

The `carpet_score` is designed to capture a vibration pattern associated with possible lubrication degradation or starved lubrication.

The idea is that lubrication-related issues can elevate broadband energy and increase the spectral floor, especially in higher frequency bands.

The score considers features such as:

- `high_frequency_energy`;
- `broadband_energy`;
- `spectral_floor_level`;
- `high_frequency_ratio`;
- `spectral_flatness`.

Conceptually:

```text
carpet_score =
    normalized combination of
    high-frequency energy
    + broadband energy
    + spectral floor level
    + high-frequency ratio
    + spectral flatness
```

A higher value indicates stronger evidence of a broadband/carpet-like pattern.

The PSD chart in the app helps visualize this behavior by showing the spectral floor and broadband energy distribution.

---

## 7. Structural looseness score

The `looseness_score` is designed to capture behavior associated with structural looseness.

The idea is that looseness often appears through low-frequency content, harmonics, subharmonics, and components related to rotational speed.

The score considers features such as:

- `low_frequency_energy`;
- `harmonic_ratio`;
- `subharmonic_ratio`;
- amplitude near `0.5x`;
- amplitude near `1x`;
- amplitude near `1.5x`;
- amplitude near `2x`;
- amplitude near `3x`.

Conceptually:

```text
looseness_score =
    normalized combination of
    low-frequency energy
    + harmonic ratio
    + subharmonic ratio
    + amplitudes around rotational frequency multiples
```

A higher value indicates stronger evidence of a looseness-like pattern.

The FFT chart in the app helps visualize this behavior by marking 0.5x, 1x, 1.5x, 2x, and 3x rotational components.

---

## 8. Overall anomaly score

The current final anomaly score is calculated as:

```python
overall_anomaly_score = max(carpet_score, looseness_score)
```

This means the final score reflects the strongest diagnostic pattern detected for that measurement.

Examples:

```text
carpet_score = 0.82
looseness_score = 0.31
overall_anomaly_score = 0.82
predicted_condition = carpet_lubrication_issue
```

```text
carpet_score = 0.28
looseness_score = 0.76
overall_anomaly_score = 0.76
predicted_condition = structural_looseness
```

If both scores are high, the measurement may be classified as:

```text
mixed_fault_pattern
```

---

## 9. Anomaly probability

The current `anomaly_probability` should be interpreted carefully.

It is not a calibrated probability from a trained statistical or machine learning model. It is a probability-like score derived from the deterministic anomaly score, for example through a sigmoid-style transformation.

A safer description is:

> `anomaly_probability` is a normalized confidence-like score derived from the deterministic anomaly score. It is not yet a calibrated ML probability.

This field is useful for visual ranking and prioritization, but it should not be presented as a statistically calibrated failure probability.

---

## 10. Severity classification

The severity is derived from the anomaly score using thresholds.

The current conceptual rule is:

```python
if overall_anomaly_score >= 0.72:
    severity = "Critical"
elif overall_anomaly_score >= 0.45:
    severity = "Attention"
else:
    severity = "Normal"
```

The Streamlit app also uses fallback logic for visual cards. It considers severity labels and risk-related fields such as:

- `anomaly_probability`;
- `overall_anomaly_score`;
- `anomaly_score`;
- average and maximum risk scores returned by SQL templates.

For consistency, future versions should centralize severity thresholds in one configuration file.

## 12. Why deterministic scoring comes before ML

Starting with deterministic scoring is intentional.

It provides:

- interpretability;
- traceability;
- simple debugging;
- business explainability;
- clear connection with physical knowledge;
- a baseline for future ML comparison;
- safer integration with GenAI and agents.

Before adding ML or deep learning, the project already answers:

- What signal was used?
- Which features were extracted?
- Which diagnostic pattern was stronger?
- Why was an asset marked as critical?
- Which action was recommended?
- What evidence can a human validate?

This is important in industrial settings, where black-box outputs without physical explanation are often difficult to trust.

## 13. Future ML roadmap

The deterministic baseline can be extended in future stages.

Possible next steps:

### 13.1 Classical ML

- Isolation Forest for unsupervised anomaly detection;
- One-Class SVM;
- Random Forest or Gradient Boosting if labeled data is available;
- clustering of vibration states;
- feature importance analysis.

### 13.2 Deep Learning

- autoencoder reconstruction error for anomaly detection;
- CNN on spectrograms or FFT windows;
- LSTM or temporal autoencoder for sequence anomaly detection;
- hybrid model combining engineered features and learned representations.

### 13.3 Model evaluation

Future ML stages should include:

- train/test split by time;
- validation strategy;
- baseline comparison against deterministic score;
- precision, recall, F1 if labels exist;
- false positive analysis;
- threshold tuning;
- confusion matrix;
- explainability layer.

### 13.4 Human validation

Future versions should include a feedback loop:

```text
model diagnosis
→ specialist review
→ accepted / corrected / rejected
→ stored validation
→ model improvement
```

This will help move from deterministic scoring to a learning system.

## 14. Recommended wording for README and interviews

Use this wording:

> The current version uses a physics-informed deterministic anomaly scoring baseline. Raw vibration signals are stored in SQLite, spectral features are extracted, and two diagnostic scores are calculated: one for carpet/lubrication behavior and one for structural looseness. The final anomaly score is the maximum of these two pattern-specific scores, and severity is assigned through explicit thresholds. This creates an interpretable baseline before introducing trained ML or deep learning models.

Avoid saying:

> The current app uses machine learning to classify failures.

A more accurate sentence is:

> The architecture is prepared for ML, but the current diagnostic layer is deterministic and physics-informed.

## 16. Current limitations

The current implementation is useful for a portfolio and architecture demonstration, but it has limitations:

- data is synthetic;
- thresholds are manually defined;
- probability is not calibrated;
- no trained ML model is active yet;
- no real maintenance labels are used;
- no expert feedback loop has been fully implemented;
- scoring thresholds need documentation and future tuning;
- severity logic should be centralized to avoid differences between seed and UI.

These limitations are acceptable at this stage as long as they are clearly communicated.
